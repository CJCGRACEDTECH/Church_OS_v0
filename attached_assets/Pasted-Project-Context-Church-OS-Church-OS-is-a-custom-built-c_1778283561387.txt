Project Context — Church OS

Church OS is a custom-built church operations platform being developed to centralize and simplify the day-to-day operational workflows of a church into one connected system.

The purpose of the project is not to build a large enterprise platform immediately. The initial goal is to create a stable, usable Version 0 (V0) that can successfully support real church operations for one church first. Once validated internally, the long-term vision is to evolve the system into a scalable multi-church SaaS platform.

The church currently relies on disconnected tools, manual workflows, spreadsheets, messaging apps, and separate systems for member tracking, attendance, children ministry operations, and giving. This creates operational friction, inconsistent data, duplicated work, and poor visibility for leadership.

Church OS is intended to become the church’s central operational system and source of truth.

Core Problems the System Solves:
- Disconnected systems and workflows
- Manual attendance tracking
- Inefficient Sunday children check-in
- Lack of centralized member and household management
- Fragmented giving workflows
- Lack of visibility into operational metrics
- Too many separate tools and logins
- Difficulty scaling church operations cleanly

Primary Goals of V0:
1. Create a stable operational foundation
2. Support real church services
3. Simplify workflows for admins and volunteers
4. Centralize member-related data
5. Enable basic giving functionality
6. Create a clean platform that can expand later

This phase is intentionally conservative and focused on stability over feature quantity.

The goal is NOT to build every feature immediately.

The goal IS to:
- create a clean architecture
- establish the core data model
- validate usability
- ensure adoption by real users
- prove the concept works in real church operations

The system currently focuses on these core operational areas:

1. Authentication & Roles
- Admin login
- Member login
- Role-based access
- Secure authentication

2. Members & Households
- Member profiles
- Household grouping
- Spouse and children relationships
- Family structure management

3. Services & Attendance
- Thursday–Sunday service support
- Attendance tracking
- Visitor tracking

4. Sunday Children Check-In
- Child check-in/check-out workflows
- Classroom assignment
- Real-time operational usability

5. Giving
- One-time giving
- Recurring tithe
- Donation tracking
- Stripe integration

6. Dashboard Visibility
- Operational summaries
- Basic metrics
- Leadership visibility

System Architecture Philosophy:
Church OS is the system of record and source of truth.

This means:
- member data lives in Church OS
- attendance data lives in Church OS
- giving data lives in Church OS
- authentication lives in Church OS

There is also a separate public-facing church website project being built in Replit.

Important:
The website is NOT the system itself.

The website acts as:
- a public-facing experience
- an informational website
- an entry point into Church OS

The website redirects users into Church OS for:
- login
- giving
- member access

The website does NOT:
- manage authentication
- store member data
- process payments independently
- duplicate Church OS functionality

Technical Philosophy:
The system is being built incrementally and conservatively.

Priorities:
1. Stability
2. Clean architecture
3. Clear ownership boundaries
4. Expandability
5. Real-world usability

The team intentionally avoids:
- overengineering
- unnecessary complexity
- building too many features at once
- large vague AI prompts
- unstable rapid feature expansion

AI Development Strategy:
The development team uses AI-assisted development carefully and intentionally.

Workflow:
- Use ChatGPT/Grok/Gemini to break down large features into smaller tasks
- Use Replit Agent only for focused implementation tasks
- Build incrementally
- Validate after each step
- Prioritize predictable outputs over speed
